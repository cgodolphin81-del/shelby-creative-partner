## 1. Community Management

### SOP-CM-001: Daily Engagement Protocol

**Purpose:** Ensure consistent, high-quality engagement across all managed communities

**Frequency:** Daily

**Owner:** Community Manager

**Procedure:**

#### Morning Routine (9:00 AM Local Time)
1. **Review Night Activity** (5 min)
   - Check #mod-logs for overnight issues
   - Review #reports for pending items
   - Scan for any escalated situations

2. **Channel Scan** (10 min)
   - Visit top 5 active channels
   - Note trending topics
   - Identify engagement opportunities

3. **Engagement Posts** (15 min)
   - Post Question of the Day
   - Share relevant resource
   - Celebrate member wins
   - Seed conversation threads

#### Afternoon Check (2:00 PM Local Time)
1. **Response Queue** (10 min)
   - Respond to all member questions
   - Engage with new introductions
   - Reply to comments on your posts

2. **Activity Boost** (10 min)
   - Post afternoon engagement content
   - Share trending topic
   - Start new discussion thread

#### Evening Wrap (6:00 PM Local Time)
1. **Day Summary** (5 min)
   - Log key activities
   - Note any issues for handoff
   - Prepare evening report

2. **Handoff Notes** (5 min)
   - Update shift log
   - Flag any pending items
   - Note upcoming events

**Success Metrics:**
- 3-5 engagement posts per day
- Response time <30 minutes
- 10+ meaningful interactions
- Zero escalated issues unresolved

**Tools:**
- Slack/Discord for communication
- Project management tool for logging
- Analytics dashboard for metrics

## 2. Moderation

### SOP-MOD-001: Moderation Shift Handoff

**Purpose:** Ensure seamless coverage between shifts

**Frequency:** Every shift change

**Procedure:**

#### Outgoing Moderator (Last 15 min of shift)
1. **Review Activity**
   - Check #mod-logs
   - Review #reports
   - Scan active channels

2. **Document Issues**
   - Active situations
   - Pending escalations
   - Members to watch
   - Channel concerns

3. **Update Handoff Log**
   ```
   SHIFT HANDOFF - [Date/Time]
   
   ACTIVE ISSUES:
   - [Issue 1] - [Status]
   - [Issue 2] - [Status]
   
   PENDING ESCALATIONS:
   - [Member] - [Issue] - [Action needed]
   
   MEMBERS TO WATCH:
   - [Member] - [Reason]
   
   CHANNEL CONCERNS:
   - [Channel] - [Issue]
   
   UPCOMING EVENTS:
   - [Event] - [Time]
   
   GENERAL NOTES:
   - [Any other relevant info]
   ```

#### Incoming Moderator (First 15 min of shift)
1. **Read Handoff Log**
   - Review all active issues
   - Note pending escalations
   - Identify priorities

2. **Quick Channel Scan**
   - Verify handoff info accurate
   - Identify new issues
   - Assess current activity level

3. **Acknowledge Handoff**
   - Post "Shift started" message
   - Confirm understanding of issues
   - Ask clarifying questions if needed

### SOP-MOD-002: Warning Procedures

**Frequency:** As needed

**Procedure:**

#### Step 1: Document Violation
- [ ] Record channel and message

- [ ] Log in moderation system

#### Step 2: Issue Warning

Hi {user},

This is a warning about your message in #{channel}.

Issue: {specific violation}

Please review our rules: {link}

may result in timeouts or bans.

#### Step 3: Log Action

- [ ] Add internal note

#### Step 4: Monitor
- [ ] Watch for repeat behavior
- [ ] Document any escalation
- [ ] Report to supervisor if pattern

**Warning Levels:**
- 1st warning: Friendly reminder
- 2nd warning: Formal warning
- 3rd warning: Final warning

### SOP-MOD-003: Timeout Procedures

**Purpose:** Ensure consistent timeout enforcement

**Frequency:** As needed

**Procedure:**

#### Step 1: Assess Severity
- [ ] First offense or repeat?
- [ ] Minor or major violation?
- [ ] Intent clear or ambiguous?
- [ ] Pattern of behavior?

#### Step 2: Choose Duration
```
Minor first offense: 1 hour
Minor repeat offense: 24 hours
Major first offense: 24 hours
Major repeat offense: 7 days
Severe violation: Permanent ban
```

#### Step 3: Execute Timeout
**Command:** `!timeout @user [duration] [reason]`

**Reason Template:**
```
{Date} - {Rule violated} - {Specific behavior}
```

#### Step 4: Notify User
**DM Template:**
```
Hi {user},

You've been timed out for {duration} due to {reason}.

This timeout will end on {date/time}.

If you believe this is an error, you may appeal at {link}.

- {community} Team
```

#### Step 5: Document
- [ ] Log in moderation system
- [ ] Update user history
- [ ] Add to escalation tracker
- [ ] Report if pattern

### SOP-MOD-005: Crisis Response

**Purpose:** Rapid, coordinated response to emergencies

**Frequency:** As needed (emergency)

**Procedure:**

#### Phase 1: Immediate Response (<1 minute)
1. **Activate Crisis Protocol**
   - Post in #mod-emergency channel
   - Alert all online mods
   - Lock server if necessary

2. **Assess Situation**
   - What type of crisis?
   - How severe?
   - Who's involved?
   - What's spreading?

#### Phase 2: Containment (1-5 minutes)
1. **Stop the Bleeding**
   - Delete harmful content
   - Timeout/ban offenders
   - Lock affected channels
   - Enable additional security

2. **Alert Team**
   - @here in mod channel
   - Assign roles
   - Coordinate response

#### Phase 3: Resolution (5-30 minutes)
1. **Resolve Issue**
   - Remove all harmful content
   - Ban all offenders
   - Restore normal operations
   - Document everything

2. **Communicate**
   - Post public statement (if needed)
   - DM affected members
   - Update leadership

#### Phase 4: Recovery (30 min - 24 hours)
1. **Restore Normal Operations**
   - Remove lockdown
   - Re-enable features
   - Monitor closely

2. **Post-Incident Review**
   - Document what happened
   - Identify what worked
   - Note improvements needed
   - Update protocols

**Crisis Types:**
- Raid attack
- Harassment campaign
- Scam/phishing
- Controversial explosion
- Staff misconduct
- Security breach

### SOP-MOD-006: Appeals Process

**Purpose:** Fair, consistent handling of ban/timeout appeals

**Frequency:** As needed

**Procedure:**

#### Step 1: Receive Appeal
- [ ] Check appeal ticket
- [ ] Review original action
- [ ] Gather all evidence
- [ ] Note appeal date

#### Step 2: Review
```
CONSIDER:
- Was action justified?
- Was response proportionate?
- Is there new information?
- Pattern of behavior?
- Member's explanation
```

#### Step 3: Decision
```
APPROVE APPEAL:
- Unban/untimeout
- Reduce duration
- Add note to file
- Notify member

DENY APPEAL:
- Maintain action
- Explain reasoning
- Note for future
- Notify member
```

#### Step 4: Communicate
**Approval Template:**
```
Hi {user},

We've reviewed your appeal and have decided to {action}.

This was based on: {reasoning}

We hope to see you back in the community!

- {community} Team
```

**Denial Template:**
```
Hi {user},

We've reviewed your appeal and have decided to {maintain action}.

This was based on: {reasoning}

If you have additional information, you may submit another 
appeal after {days} days.

- {community} Team
```

#### Step 5: Document
- [ ] Record decision
- [ ] Note reasoning
- [ ] Update user history
- [ ] Report if unusual case

### SOP-TECH-003: Permission Audit

**Purpose:** Regular review of server permissions for security

**Frequency:** Monthly

**Procedure:**

#### Step 1: Gather Data
- [ ] Export role list
- [ ] Export channel permissions
- [ ] List all bots and permissions
- [ ] Document custom permissions

#### Step 2: Review Each Role
```
CHECK:
✅ Does role need this permission?
✅ Can members grant this to themselves?
✅ Is permission appropriate for role level?
✅ Any dangerous permissions?
```

**Dangerous Permissions:**
- Administrator (avoid entirely)
- Manage Server (limit to owners)
- Manage Roles (limit to admins)
- Manage Webhooks (review carefully)
- Kick/Ban (limit to mods+)

#### Step 3: Identify Issues
- Overly permissive roles
- Members with admin access
- Orphaned roles
- Duplicate permissions
- Missing restrictions

#### Step 4: Fix Issues
- [ ] Remove unnecessary permissions
- [ ] Restrict dangerous permissions
- [ ] Update role hierarchy
- [ ] Document changes
- [ ] Test in private channel

#### Step 5: Document
- [ ] Create permission matrix
- [ ] Update documentation
- [ ] Report findings
- [ ] Schedule next audit

### SOP-TECH-005: Backup Procedures

**Purpose:** Ensure server data can be recovered

**Frequency:** Daily automatic, monthly manual verification

**Procedure:**

#### Automatic Backups
1. **Enable Backup Bot**
   - Set daily schedule (3 AM UTC)
   - Include all channels
   - Include all roles
   - Include emojis and stickers

2. **Configure Retention**
   - Keep 30 days of daily backups
   - Keep 12 months of monthly backups
   - Store in secure location

#### Manual Verification
1. **Monthly Test**
   - Restore from backup
   - Verify data integrity
   - Test all features
   - Document results

2. **Emergency Backup**
   - Export channel data
   - Save role hierarchy
   - Document permissions
   - Store externally

#### Recovery Procedure
1. **Assess Damage**
   - What was lost?
   - What needs restoring?
   - Which backup to use?

2. **Execute Recovery**
   - Restore channels
   - Restore roles
   - Restore permissions
   - Test functionality

3. **Verify**
   - Check all channels
   - Verify permissions
   - Test bots
   - Confirm access

### SOP-GA-003: A/B Testing

**Purpose:** Data-driven optimization

**Frequency:** Ongoing

**Procedure:**

#### Test Setup
1. **Define Hypothesis**
   - What are we testing?
   - Expected outcome?
   - Success metrics?

2. **Create Variants**
   - Version A (control)
   - Version B (variant)
   - Keep one variable different

3. **Run Test**
   - Split audience
   - Run for sufficient time
   - Track metrics

4. **Analyze Results**
   - Statistical significance
   - Practical significance
   - Learnings

5. **Implement**
   - Roll out winner
   - Document learnings
   - Plan next test
